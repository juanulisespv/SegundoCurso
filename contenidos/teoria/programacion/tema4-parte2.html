Diagramas de Comportamiento UML

Los diagramas de comportamiento muestran cómo funciona el sistema - las interacciones, flujos y cambios de estado.

Diagrama de Secuencia

Interacciones en el Tiempo

Muestra cómo los objetos colaboran para realizar una funcionalidad

Ejemplo: Login de usuario
Usuario    │ Sistema   │ BD
   │        │           │
   ├────────▶ login()   │
   │        ├───────────▶ verificar()
   │        ◁───────────┤ true/false
   │        │           │
   ◁────────┤ sesión    │

Diagrama de Actividad

Flujos de Proceso

Representa workflow y lógica de negocio con decisiones y paralelismo

Ejemplo: Proceso de compra
○ Inicio
↓
□ Seleccionar productos
↓
◇ ¿Carrito vacío? → Sí → ○ Fin
↓ No
□ Proceso pago
↓
□ Confirmar orden
↓
○ Fin

Diagrama de Estados

Ciclos de Vida

Modela estados de objetos y transiciones entre ellos

Estados de un pedido:
Pendiente → [pago confirmado] → Procesando
     ↓                              ↓
Cancelado ← [timeout/error] ← Enviado
                              ↓
                          Entregado

Cuándo Usar Cada Diagrama

  • Secuencia: Colaboración entre objetos específicos
  • Actividad: Procesos de negocio y workflows complejos
  • Estados: Objetos con ciclos de vida definidos

Modelado de Datos

El modelado de datos diseña cómo se almacenan, organizan y relacionan los datos en el sistema.

Modelo Entidad-Relación (ER)

Componentes del Modelo ER

Entidades: Objetos del mundo real

Usuario, Producto, Pedido

Atributos: Propiedades de entidades

nombre, precio, fecha

Relaciones: Conexiones entre entidades

Cliente "realiza" Pedido

Ejemplo de modelo ER:
USUARIO (id, nombre, email, fecha_registro)
   │
   │ 1:N (realiza)
   ▼
PEDIDO (id, fecha, total, estado, usuario_id)
   │
   │ N:M (contiene)
   ▼
PRODUCTO (id, nombre, precio, stock)

Normalización de Bases de Datos

Primera Forma Normal (1NF)

Eliminar valores repetidos

Cada celda contiene un solo valor

Segunda Forma Normal (2NF)

Eliminar dependencias parciales

Todos los atributos dependen de la clave completa

Tercera Forma Normal (3NF)

Eliminar dependencias transitivas

Atributos no clave no dependen entre sí

Normalización vs. Rendimiento

La normalización reduce redundancia pero puede afectar el rendimiento. En sistemas con muchas consultas, a veces se desnormaliza intencionalmente.

Ejemplo de normalización:
Sin normalizar (problemático):
PEDIDO (id, cliente_nombre, cliente_email, producto_nombre, precio)

Normalizado (correcto):
CLIENTE (id, nombre, email)
PRODUCTO (id, nombre, precio)  
PEDIDO (id, cliente_id, producto_id, cantidad)

Patrones de Diseño

Los patrones de diseño son soluciones reutilizables a problemas comunes en el diseño de software.

¿Por qué Usar Patrones?

  • Experiencia probada: Soluciones validadas
  • Comunicación: Vocabulario común entre desarrolladores
  • Reutilización: No reinventar la rueda
  • Mantenibilidad: Código más organizado

Patrones Fundamentales

Singleton

Una sola instancia global

Para configuración, logs, conexiones BD

class ConfigManager:
    _instance = None
    
    def __new__(cls):
        if not cls._instance:
            cls._instance = super().__new__(cls)
        return cls._instance
Factory

Crear objetos sin especificar clase exacta

Para crear diferentes tipos de objetos

class VehiculoFactory:
    def crear(tipo):
        if tipo == "auto":
            return Auto()
        elif tipo == "moto":
            return Moto()
Observer

Notificar cambios automáticamente

Para actualizaciones de UI, eventos

class Observable:
    def __init__(self):
        self.observers = []
    
    def notify(self):
        for obs in self.observers:
            obs.update()
Strategy

Algoritmos intercambiables

Para diferentes comportamientos

class PagoContext:
    def __init__(self, strategy):
                        self.strategy = strategy
    
    def procesar(self, amount):
        return self.strategy.pagar(amount)

Cuándo NO Usar Patrones

No aplicar patrones "porque sí". Úsalos solo cuando resuelvan un problema específico y mejoren el diseño.

Mejores Prácticas en Modelado

Seguir mejores prácticas asegura que los modelos sean útiles, mantenibles y comprensibles.

Principios de Modelado Efectivo

Simplicidad

Modelos simples comunican mejor

  • Un diagrama, un propósito
  • No más de 7±2 elementos por vista
  • Niveles de abstracción apropiados
Orientado a Audiencia

Modelos para diferentes stakeholders

  • Clientes: casos de uso
  • Desarrolladores: clases detalladas
  • Arquitectos: componentes de alto nivel
Evolutivos

Modelos que evolucionan

  • Actualizados con el código
  • Versionados en control de código
  • Refinados iterativamente
Validados

Verificados con stakeholders

  • Revisiones con usuarios
  • Validación técnica
  • Pruebas de concepto

Errores Comunes en Modelado

  • Sobre-modelado: Crear diagramas que nadie mantiene
  • Modelos obsoletos: No actualizar con cambios de código
  • Demasiado detalle: Diagramas incomprensibles
  • Falta de validación: Modelos que no reflejan realidad

Metodologías de Modelado

Domain-Driven Design (DDD)

Enfoque en el Dominio

El modelo debe reflejar el lenguaje y conceptos del negocio, no solo la implementación técnica.

Lenguaje Ubicuo

Vocabulario común

Desarrolladores y expertos del negocio usan los mismos términos

Modelos de Dominio

Entidades del mundo real

Objetos que tienen identidad y comportamiento en el negocio

Model-Driven Development (MDD)

De Modelos a Código

Generar código automáticamente desde modelos de alto nivel

Flujo MDD:
Modelo UML → Generador → Código Base → Desarrollo Manual

Ventajas y Desafíos del MDD

Ventajas:

  • Desarrollo más rápido
  • Menos errores de codificación
  • Documentación automática

Desafíos:

  • Herramientas costosas
  • Curva de aprendizaje
  • Pérdida de control del código

Integración del Modelado en el Desarrollo

El modelado debe estar integrado en el proceso de desarrollo, no ser una actividad aislada.

Modelado Ágil

Principios del Modelado Ágil

  • Just enough: Modelar solo lo necesario
  • Just in time: Modelar cuando se necesite
  • Colaborativo: Toda el equipo participa
  • Iterativo: Refinar modelos constantemente
Modelado Continuo
  • Sesiones de diseño regulares
  • Modelos actualizados con refactoring
  • Documentación viviente
Modelado Colaborativo
  • Sesiones de whiteboarding
  • Revisiones de diseño
  • Pair modeling
Integración en Sprints:
Sprint Planning: Modelar historias complejas
Durante desarrollo: Refinar modelos según necesidad
Sprint Review: Validar modelos con stakeholders
Retrospectiva: Evaluar efectividad del modelado
