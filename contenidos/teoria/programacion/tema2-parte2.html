Scrum Framework

Scrum es el framework ágil más popular, que organiza el trabajo en ciclos cortos llamados Sprints con roles, eventos y artefactos bien definidos.

Los 3 Roles de Scrum

Cada rol tiene responsabilidades específicas y complementarias

Product Owner

El "qué" del producto

  • Define el Product Backlog
  • Prioriza funcionalidades
  • Acepta o rechaza entregables
  • Representa la voz del cliente
Development Team

El "cómo" del producto

  • Auto-organizado (3-9 personas)
  • Cross-funcional
  • Entrega software funcionando
  • Estima y planifica su trabajo
Scrum Master

El facilitador del proceso

  • Facilita eventos Scrum
  • Elimina impedimentos
  • Protege al equipo
  • Enseña prácticas ágiles

Sprint: El Corazón de Scrum

Un Sprint es un período de tiempo fijo (1-4 semanas) donde el equipo crea un incremento de producto potencialmente liberable.

Eventos Scrum

Scrum define 5 eventos principales que proporcionan estructura y oportunidades de inspección y adaptación.

Sprint Planning

Duración: 8h para Sprint de 4 semanas

  • ¿Qué se puede entregar?
  • ¿Cómo se hará el trabajo?
  • Crear Sprint Backlog
Daily Scrum

Duración: 15 minutos máximo

  • ¿Qué hice ayer?
  • ¿Qué haré hoy?
  • ¿Qué impedimentos tengo?
Sprint Review

Duración: 4h para Sprint de 4 semanas

  • Demostrar trabajo completado
  • Obtener feedback del Product Owner
  • Adaptar Product Backlog
Sprint Retrospective

Duración: 3h para Sprint de 4 semanas

  • ¿Qué salió bien?
  • ¿Qué se puede mejorar?
  • Plan de mejora para el próximo Sprint
Ejemplo de cronograma Sprint de 2 semanas:
Lunes Semana 1: Sprint Planning (4h)
Lunes-Viernes: Daily Scrum (15min diarios)
Viernes Semana 2: Sprint Review (2h) + Retrospective (1.5h)

Metodología Kanban

Kanban se centra en la visualización del flujo de trabajo y la mejora continua, sin prescribir roles o eventos específicos.

Principios Fundamentales de Kanban

Visualizar el Trabajo

Hacer visible todo el trabajo y su estado

Limitar WIP

Work In Progress limitado mejora el flujo

Gestionar el Flujo

Optimizar la velocidad de entrega

Mejora Continua

Experimentar y evolucionar constantemente

Tablero Kanban Típico

Backlog

Ideas y tareas pendientes

En Progreso

Trabajo activo (WIP: 3)

En Revisión

Esperando feedback

Completado

Tareas terminadas

Ventaja de Kanban

Kanban es más flexible que Scrum y se puede implementar gradualmente sobre procesos existentes.

DevOps: Desarrollo + Operaciones

DevOps es una cultura y conjunto de prácticas que une el desarrollo de software (Dev) con las operaciones de TI (Ops).

Objetivos de DevOps

  • Acelerar la entrega de software
  • Mejorar la calidad y confiabilidad
  • Automatizar procesos repetitivos
  • Fomentar la colaboración entre equipos

Prácticas Clave de DevOps

Integración Continua (CI)

Mergear código frecuentemente con testing automático

git push → tests automáticos → merge si todo pasa
Despliegue Continuo (CD)

Automatizar el despliegue a producción

código aprobado → deploy automático → monitoreo
Infrastructure as Code

Gestionar infraestructura mediante código versionado

Terraform, Docker, Kubernetes para configurar servidores
Monitoring y Observabilidad

Monitorear aplicaciones en tiempo real

Logs, métricas, alertas para detectar problemas rápidamente

Metodologías Híbridas

Las organizaciones modernas combinan elementos de diferentes metodologías para crear enfoques personalizados que se adapten a sus necesidades específicas.

Ejemplos de Combinaciones Populares

Scrum + Kanban (Scrumban)
  • Sprints de Scrum para ritmo
  • Tablero visual de Kanban
  • Límites WIP para optimizar flujo
  • Retrospectivas para mejora continua
Agile + Waterfall
  • Requisitos definidos inicialmente
  • Desarrollo iterativo dentro de fases
  • Testing continuo durante desarrollo
  • Entrega incremental de funcionalidades
DevOps + Scrum
  • Sprints para organizar desarrollo
  • CI/CD pipeline para automatización
  • Monitoring integrado en Definition of Done
  • Colaboración Dev-Ops en el equipo
SAFe (Scaled Agile)
  • Múltiples equipos Scrum coordinados
  • Program Increment (8-12 semanas)
  • Arquitectura y planning a gran escala
  • Value Streams end-to-end

¿Cómo Elegir la Metodología Correcta?

La elección depende de múltiples factores del proyecto, equipo y organización. No existe una metodología perfecta para todos los casos.

Factor Waterfall Scrum Kanban
Tamaño del Equipo Grande (10+ personas) Mediano (3-9 personas) Cualquier tamaño
Estabilidad de Requisitos Muy estables Moderadamente cambiantes Altamente cambiantes
Complejidad del Proyecto Baja-Media Media-Alta Media
Experiencia del Equipo Baja en agilidad Media-Alta en colaboración Alta en auto-gestión
Frecuencia de Entrega Una gran entrega Cada 1-4 semanas Continua cuando esté listo

Guía de Decisión

  • Waterfall: Proyectos con requisitos fijos y regulaciones estrictas
  • Scrum: Productos nuevos con equipos colaborativos
  • Kanban: Mantenimiento o mejoras continuas
  • Híbrida: Organizaciones grandes con múltiples restricciones

Conclusión del Tema 2

Las metodologías de desarrollo son herramientas al servicio del equipo, no reglas absolutas que limiten la creatividad.

Puntos Clave del Tema

  • Waterfall funciona bien con requisitos estables y proyectos predecibles
  • Agile es superior para productos innovadores y equipos colaborativos
  • Scrum proporciona estructura con flexibilidad
  • Kanban optimiza el flujo de trabajo continuo
  • DevOps integra desarrollo con operaciones
  • Enfoques híbridos combinan lo mejor de cada metodología
