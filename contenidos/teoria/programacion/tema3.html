1. Introducción

Definición

La ingeniería de requisitos es el proceso sistemático y estructurado de identificar, analizar, documentar, verificar y gestionar las necesidades y expectativas de los stakeholders respecto a un sistema o producto.

Su objetivo es garantizar que el software desarrollado cumpla con los requisitos funcionales (qué debe hacer el sistema) y no funcionales (cómo debe comportarse), así como con las restricciones técnicas y normativas.

Importancia en el Ciclo de Vida

Los requisitos son el pilar fundamental en el ciclo de vida del software, ya que establecen qué necesita y espera el cliente del sistema. Una definición clara y precisa asegura que todas las etapas posteriores se realicen alineadas con los objetivos del proyecto.

Beneficios
  • Minimiza riesgos y malentendidos
  • Optimiza recursos y tiempo
  • Mejora calidad del producto
  • Facilita validación y pruebas
Riesgos de Falta Claridad
  • Errores costosos en etapas avanzadas
  • Retrasos en la entrega
  • Productos que no satisfacen al cliente
  • Conflictos entre stakeholders

Objetivos y Alcance

Definir Necesidades
Garantizar que todos los stakeholders comprendan los objetivos
Resolver Conflictos
Detectar inconsistencias entre requisitos y expectativas
Base Sólida
Marco que guíe diseño, implementación y pruebas
Gestionar Cambios
Adaptarse sin comprometer calidad ni plazos

2. Tipos de Requisitos

Los requisitos se clasifican en funcionales, no funcionales, técnicos y legales, abarcando desde las funcionalidades del software hasta las características de rendimiento y normativas.

2.1. Requisitos Funcionales

Describen las funcionalidades específicas que un sistema debe realizar para cumplir con las necesidades del usuario. Definen qué debe hacer el software.

Ejemplos Prácticos:
  1. El sistema debe permitir a los usuarios registrarse y autenticar su cuenta
  2. Los usuarios podrán buscar productos por categoría y añadirlos a un carrito
  3. El software debe generar reportes mensuales de ventas en formato PDF
  4. El sistema debe enviar notificaciones por correo electrónico
Casos de Uso:
Caso de Uso: Iniciar Sesión Actor: Usuario registrado Flujo Principal: 1. Usuario ingresa credenciales 2. Sistema valida autenticación 3. Usuario accede a su perfil Excepciones: - Credenciales incorrectas - Cuenta bloqueada

2.2. Requisitos No Funcionales

Describen las características de calidad que debe cumplir el sistema. Definen cómo debe comportarse y bajo qué condiciones debe operar.

Rendimiento

"El sistema debe procesar 100 transacciones por segundo con tiempo de respuesta inferior a 2 segundos"

Seguridad

"Los datos sensibles deben encriptarse con estándar AES de 256 bits"

Usabilidad

"Debe ser accesible desde móviles y cumplir normas WCAG 2.1"

Escalabilidad

"Debe soportar hasta 10,000 usuarios concurrentes sin degradar rendimiento"

2.3. Requisitos Técnicos

Especifican las condiciones tecnológicas necesarias para que el software funcione correctamente, estableciendo el marco técnico de desarrollo y operación.

Dependencias Tecnológicas:
  • Lenguaje: Python 3.10 o superior
  • Base de datos: PostgreSQL
  • Servidor: AWS EC2
  • Integración: API Stripe para pagos
Especificaciones del Sistema:
  • Hardware: 16 GB RAM, 8 núcleos
  • SO: Linux Ubuntu 22.04
  • Red: Mínimo 10 Mbps
  • Navegadores: Chrome, Firefox, Safari

2.4. Requisitos Legales y Normativos

Garantizan que el sistema cumpla con leyes, regulaciones y estándares aplicables. Su incumplimiento puede resultar en sanciones legales y pérdida de confianza.

RGPD
Protección de datos personales en Europa
WCAG/ADA
Accesibilidad para personas con discapacidades
PCI-DSS
Seguridad para transacciones financieras

3. Fases de la Ingeniería de Requisitos

Proceso estructurado que incluye recolección, análisis, especificación y validación para minimizar riesgos y garantizar alineación con objetivos del proyecto.

3.1. Recolección

Primera fase que identifica y captura las necesidades, expectativas y restricciones de los stakeholders. Una recolección incompleta puede llevar a malentendidos durante el desarrollo.

Técnicas de Recopilación:
Entrevistas
Reuniones individuales/grupales estructuradas o abiertas
Encuestas
Recopilar datos de gran número de personas
Talleres
Sesiones colaborativas para alineación
Herramientas de Gestión:
  • User Stories: "Como estudiante, quiero acceder a mis calificaciones para conocer mi progreso"
  • Prototipos: Maquetas que permiten visualizar funcionamiento
  • Mapas Mentales: Organizan ideas y muestran relaciones entre conceptos

3.2. Análisis

Examina la información recopilada para identificar inconsistencias, conflictos o redundancias y priorizar los requisitos según las necesidades del proyecto y stakeholders.

Identificación de Conflictos:

Resolver discrepancias entre stakeholders. Ejemplo: Un usuario solicita simplicidad mientras otro exige funcionalidades avanzadas que complican el diseño.

Técnicas de Priorización:
MoSCoW
  • Must: Imprescindible
  • Should: Importante
  • Could: Deseable
  • Won't: No incluir
Análisis de Valor

Evalúa impacto vs costo de implementación de cada requisito

Ranking

Stakeholders asignan nivel de prioridad en escala definida

3.3. Especificación

Documentar de manera clara, precisa y estructurada las necesidades y características del sistema. Sirve como referencia para equipo técnico y stakeholders.

Estándares y Plantillas:
IEEE 830

Estándar para Documento de Especificación de Requisitos de Software (SRS)

Plantillas Personalizadas

Adaptadas a necesidades específicas del proyecto o cliente

Estructura del Documento:
  1. Introducción: Objetivos, alcance y definiciones clave
  2. Descripción general: Contexto, actores y dependencias
  3. Requisitos funcionales: Funcionalidades detalladas
  4. Requisitos no funcionales: Características de calidad
  5. Restricciones: Normativas aplicables

3.4. Validación

Asegura que las necesidades y expectativas estén correctamente representadas y documentadas. Verifica que los requisitos sean claros, consistentes, completos y alcanzables.

Métodos de Verificación:
Revisiones
Reuniones con stakeholders
Prototipos
Modelos preliminares
Casos de Prueba
Verificación específica
Trazabilidad
Vinculación a objetivos

4. Herramientas y Técnicas

4.1. Herramientas Digitales

Jira

Gestión de proyectos ágiles. Organiza requisitos como tickets/historias de usuario, asignación a equipos y seguimiento con tableros Kanban/Scrum.

Trello

Herramienta visual con tableros y tarjetas. Ideal para proyectos pequeños que buscan solución simple para organizar y priorizar tareas.

Herramientas Especializadas:
DOORS

Para proyectos complejos (ingeniería, automoción). Documentar, rastrear y gestionar requisitos detallados.

ReqView

Centrada en trazabilidad. Mantiene relación clara entre requisitos y entregables del proyecto.

4.2. Técnicas Visuales

Casos de Uso

Muestran cómo usuarios (actores) interactúan con el sistema. Incluyen actores, casos de uso y relaciones.

Diagramas UML

Clases, secuencia, actividades. Representan diferentes aspectos del sistema y sus interacciones.

Mapas Mentales

Organizan ideas jerárquicamente mostrando módulos, funcionalidades y dependencias.

4.3. Prototipado

Crear representaciones preliminares del sistema para clarificar requisitos ofreciendo visión tangible del producto final.

Maquetas

Representaciones estáticas de estructura y diseño. Herramientas: Figma, Adobe XD.

Prototipos Funcionales

Versiones interactivas simplificadas. Herramientas: Axure, desarrollo rápido.

Ventajas del Prototipado:
  • Detecta y corrige errores en primeras etapas
  • Facilita retroalimentación de stakeholders
  • Reduce riesgos y costos de cambios tardíos

5. Desafíos y Soluciones

5.1. Gestión de Cambios

Proceso crítico que asegura que las modificaciones durante el desarrollo sean evaluadas, aprobadas y documentadas adecuadamente.

Pasos de Gestión:
1
Identificación
Documentar el cambio propuesto y sus razones
2
Evaluación
Analizar impacto en alcance, costo, tiempo
3
Aprobación
Revisión por comité o stakeholders
4
Implementación
Incorporar y actualizar documentos

5.2. Comunicación con Stakeholders

La comunicación efectiva garantiza que todas las partes comprendan y acuerden las necesidades del sistema, adaptando el enfoque según roles y perspectivas.

Patrocinador

Enfoque: Objetivos estratégicos, ROI, plazos
Comunicación: Resultados generales y avances

Director Funcional

Enfoque: Procesos de negocio, expectativas operativas
Comunicación: Alineación con negocio

Usuario Directo

Enfoque: Funcionalidad, accesibilidad, usabilidad
Comunicación: Demostraciones y prototipos

5.3. Equilibrio entre Requisitos

Lograr equilibrio entre requisitos funcionales (qué debe hacer) y no funcionales (cómo debe comportarse) es fundamental para el éxito del proyecto.

Importancia del Equilibrio:

Solo funcionales: Sistema que funciona pero no es eficiente, seguro o usable

Solo no funcionales: Sistema optimizado pero carece de características fundamentales

Estrategias:
  1. Priorización conjunta: Usar MoSCoW para ambos tipos
  2. Evaluación iterativa: Revisar cómo se afectan mutuamente
  3. Validación temprana: Pruebas que aseguren rendimiento

Casos Prácticos

Sistema de Citas Médicas
Requisitos Funcionales:
  • Registro y autenticación de pacientes
  • Selección de médico y reserva de cita
  • Notificaciones por email
  • Consulta historial de citas
Requisitos No Funcionales:
  • 500 solicitudes/segundo
  • Cumplimiento RGPD
  • Accesibilidad WCAG 2.1
Tienda Online
Requisitos Funcionales:
  • Registro y acceso de clientes
  • Navegación por categorías
  • Pagos seguros con pasarela
  • Confirmación por email
Requisitos No Funcionales:
  • 1000 pedidos simultáneos
  • Carga página < 2 segundos
  • Cumplimiento RGPD
