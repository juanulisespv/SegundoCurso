Documentación de Requisitos

La documentación efectiva es fundamental para comunicar requisitos de manera clara y sin ambigüedades.

Especificación de Requisitos de Software (SRS)

Documento SRS

Documento formal que define requisitos funcionales y no funcionales

Estructura típica de SRS:
1. Introducción y propósito
2. Descripción general del producto
3. Requisitos funcionales específicos
4. Requisitos no funcionales
5. Interfaces externas
6. Restricciones y limitaciones
Características SRS
  • Completo: Toda funcionalidad
  • Consistente: Sin contradicciones
  • Inequívoco: Una sola interpretación
  • Verificable: Se puede probar
Audiencia del SRS
  • Clientes: Validan necesidades
  • Desarrolladores: Guía implementación
  • Testers: Base para pruebas
  • Gestores: Planificación

Casos de Uso

Los casos de uso describen interacciones específicas entre actores (usuarios) y el sistema para lograr un objetivo.

Componentes de un Caso de Uso

  • Actor: Quien inicia la interacción
  • Objetivo: Qué quiere lograr
  • Precondiciones: Estado inicial
  • Flujo principal: Pasos normales
  • Flujos alternativos: Variaciones
  • Postcondiciones: Estado final
Ejemplo de Caso de Uso:
Caso de Uso: Realizar Compra Online
Actor: Cliente registrado
Objetivo: Comprar productos y recibirlos

Flujo Principal:
1. Cliente busca productos
2. Añade items al carrito
3. Revisa carrito y procede al checkout
4. Ingresa datos de envío y pago
5. Confirma orden
6. Sistema procesa pago
7. Sistema confirma orden por email
Flujos Alternativos
  • Pago rechazado: volver a paso 4
  • Producto agotado: notificar y remover
  • Usuario cancela: vaciar carrito
Flujos de Excepción
  • Error de conexión: mostrar mensaje
  • Sesión expirada: redirigir a login
  • Sistema no disponible: página mantenimiento

Historias de Usuario

Las historias de usuario son requisitos ágiles escritos desde la perspectiva del usuario final, enfocándose en el valor de negocio.

Formato Estándar

"Como [tipo de usuario], quiero [funcionalidad], para [beneficio]"

Ejemplos de Historias de Usuario:
Historia 1:
Como cliente frecuente, quiero guardar mis métodos de pago, 
para completar compras más rápidamente.

Historia 2:
Como administrador, quiero ver reportes de ventas diarias, 
para tomar decisiones de inventario.

Historia 3:
Como usuario móvil, quiero usar la aplicación offline, 
para acceder a mi información sin conexión.

Criterios de Aceptación

Definiendo "Terminado"

Los criterios de aceptación especifican cuándo una historia está completa y funcionando correctamente.

Criterios para "Guardar métodos de pago":
  • ✓ Usuario puede añadir tarjetas de crédito/débito
  • ✓ Datos se almacenan encriptados
  • ✓ Usuario puede eliminar métodos guardados
  • ✓ Funciona en checkout con un click
  • ✓ Compatible con regulaciones PCI-DSS

Validación y Verificación de Requisitos

Es crucial asegurar que los requisitos son correctos, completos y consistentes antes de implementar.

Verificación

"¿Estamos construyendo el producto correctamente?"

Comprueba que los requisitos cumplan estándares de calidad

Validación

"¿Estamos construyendo el producto correcto?"

Confirma que los requisitos satisfacen necesidades reales

Técnicas de Verificación

Revisiones Formales
  • Inspecciones de código/documentos
  • Walkthroughs guiados
  • Revisiones por pares
Checklists
  • Verificar completitud
  • Detectar inconsistencias
  • Validar trazabilidad

Técnicas de Validación

Prototipos
  • Mockups de interfaz
  • Prototipos funcionales
  • Demos tempranas
Validación con Usuarios
  • Testing de usabilidad
  • Sesiones de feedback
  • Pruebas piloto

Errores Comunes

  • Validar solo con stakeholders técnicos - ignorar usuarios finales
  • Verificación tardía - revisar solo al final del proyecto
  • Documentación sin validación - asumir que escribir = correcto

Gestión de Cambios en Requisitos

Los requisitos evolucionan durante el proyecto. Una gestión efectiva de cambios es esencial para el éxito.

Realidad de los Cambios

En proyectos típicos, entre 25-50% de los requisitos cambian durante el desarrollo.

Proceso de Gestión de Cambios

Solicitud

Identificar la necesidad de cambio

  • Formulario estándar
  • Justificación del cambio
  • Impacto estimado
Evaluación

Analizar costo vs. beneficio

  • Impacto técnico
  • Costo de implementación
  • Efecto en cronograma
Decisión

Aprobar o rechazar con justificación

  • Comité de cambios
  • Criterios claros
  • Comunicación de decisión

Trazabilidad de Requisitos

Mantener vínculos entre requisitos, diseño, código y pruebas permite:

  • Análisis de impacto de cambios
  • Validación de cobertura en testing
  • Justificación de decisiones de diseño
Matriz de trazabilidad simplificada:
REQ-001: Login usuario → UC-001, TEST-001-005
REQ-002: Búsqueda productos → UC-002, TEST-006-012  
REQ-003: Proceso pago → UC-003, TEST-013-020

Herramientas para Gestión de Requisitos

Las herramientas adecuadas facilitan la captura, documentación y gestión de requisitos durante todo el ciclo de vida.

Documentación
  • Confluence: Wiki colaborativo
  • Notion: Documentos estructurados
  • Google Docs: Colaboración simple
Gestión Ágil
  • Jira: Historias y backlog
  • Azure DevOps: Gestión completa
  • Trello: Kanban simple
Modelado
  • Lucidchart: Diagramas UML
  • Draw.io: Diagramas gratuitos
  • Enterprise Architect: Modelado empresarial
Prototipado
  • Figma: Prototipos interactivos
  • InVision: Mockups colaborativos
  • Balsamiq: Wireframes rápidos

Selección de Herramientas

Criterios de Selección

  • Tamaño del equipo: Herramientas simples vs. empresariales
  • Metodología: Ágil (Jira) vs. Tradicional (DOORS)
  • Integración: Compatibilidad con herramientas existentes
  • Presupuesto: Opciones gratuitas vs. licencias comerciales
Stack típico para equipos pequeños:
• Documentación: Notion/Confluence
• Gestión: Jira/Trello  
• Diagramas: Draw.io/Lucidchart
• Prototipos: Figma/Balsamiq
• Comunicación: Slack/Teams

Mejores Prácticas

  • Integración: Herramientas que se conecten entre sí
  • Simplicidad: No sobre-complicar el proceso
  • Adopción: Entrenar al equipo en las herramientas seleccionadas
  • Respaldo: Versiones y backups de documentación

Resumen del Tema 3

Conceptos Clave Aprendidos

  • Tipos de requisitos (funcionales/no funcionales)
  • Identificación de stakeholders
  • Técnicas de elicitación
  • Documentación efectiva (SRS, casos de uso, historias)
  • Validación y verificación

La ingeniería de requisitos es un proceso iterativo que requiere comunicación constante, documentación clara y gestión proactiva de cambios.
