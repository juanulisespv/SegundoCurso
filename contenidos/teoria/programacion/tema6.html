Introducción al Modelado Conceptual Avanzado

El modelado conceptual avanzado es una metodología estructurada para representar sistemas complejos usando principios de POO.

¿Qué es el Modelado Conceptual Avanzado?

Metodología para representar de manera estructurada y detallada los componentes del sistema, sus relaciones y comportamientos.

  • Se basa en principios de POO
  • Fundamental para aplicaciones escalables
  • Promueve reutilización y mantenibilidad

Importancia en el Diseño de Software

Claridad Estructural

Visualizar jerarquías y relaciones

  • Estructura del sistema clara
  • Relaciones entre elementos
  • Organización antes de implementar
Colaboración Facilitada

Representación común del sistema

  • Diseñadores y desarrolladores
  • Analistas y arquitectos
  • Lenguaje común de comunicación
Mantenimiento y Escalabilidad

Software evoluciona fácilmente

  • Diseño bien estructurado
  • Evolución sin comprometer funcionalidad
  • Adaptabilidad a cambios
Reutilización de Código

Clases y relaciones reutilizables

  • Definición correcta de clases
  • Reutilización en futuros proyectos
  • Eficiencia en desarrollo

Relación con la Programación Orientada a Objetos

Principios Fundamentales Compartidos
  • Herencia: Clases derivan de otras, reutilizando código
  • Polimorfismo: Métodos genéricos, implementaciones específicas
  • Encapsulamiento: Acceso controlado a atributos
  • Composición: Relaciones sin herencia, mayor modularidad

Beneficios en Sistemas Complejos

Reutilización de Código: Herencia y composición optimizan tiempo y recursos

Flexibilidad y Escalabilidad: Sistemas adaptan y crecen sin grandes modificaciones

Herencia en el Modelado Conceptual

La herencia es un principio clave de POO que permite a una clase derivar de otra, heredando atributos y métodos.

Conceptos Fundamentales

Principios Básicos de la Herencia

  1. Reutilización de código: Subclases reutilizan atributos y métodos, evitando redundancias
  2. Jerarquía de clases: Estructura clara, clases generales arriba, especializadas abajo
  3. Extensibilidad: Añadir funcionalidades sin modificar superclase
  4. Relación "es un": Subclase es tipo más específico de superclase
Ejemplo básico en Python:
class Animal:
    def __init__(self, nombre):
        self.nombre = nombre
    
    def hacer_sonido(self):
        return "Sonido genérico"

class Perro(Animal):  # Hereda de Animal
    def hacer_sonido(self):
        return "Ladrido"

mi_perro = Perro("Firulais")
print(mi_perro.hacer_sonido())  # Salida: Ladrido

Representación en UML

+------------------+
|      Animal      |
+------------------+
| - nombre: String |
+------------------+
| + hacer_sonido() |
+------------------+
         ▲
         │ (hereda)
+------------------+
|      Perro       |
+------------------+
| + hacer_sonido() |
+------------------+

Sobreescritura de Métodos

Override vs Overload

Sobreescritura (Override):

  • Subclase redefine método padre
  • Mismo nombre y firma
  • Comportamiento específico

Sobrecarga (Overload):

  • Múltiples métodos mismo nombre
  • Diferentes parámetros
  • Python: simulado con defaults
Uso de super() en Python:
class Animal:
    def hacer_sonido(self):
        return "Sonido genérico"

class Perro(Animal):
    def hacer_sonido(self):
        sonido_base = super().hacer_sonido()  # Llamada al padre
        return f"{sonido_base} - Ladrido"

mi_perro = Perro()
print(mi_perro.hacer_sonido())  # Salida: Sonido genérico - Ladrido

Ventajas y Desventajas de la Herencia

Usar herencia cuando: Existe relación "es un", clases comparten atributos, se requiere polimorfismo

Alternativas: Composición para relaciones "tiene un", interfaces para contratos, delegación para responsabilidades

Polimorfismo y su Importancia en el Diseño

El polimorfismo permite que diferentes clases respondan de manera distinta a un mismo método, mejorando flexibilidad y escalabilidad.

Tipos de Polimorfismo

Por Sobreescritura (Override)

Subclase redefine método padre

  • Usado en herencia
  • Comportamiento específico por clase
  • Mismo nombre, implementación diferente
Por Sobrecarga (Overloading)

Múltiples métodos mismo nombre

  • Diferentes parámetros
  • Python: valores por defecto
  • Java/C++: sobrecarga estricta
Ejemplo polimorfismo en acción:
class Animal:
    def hacer_sonido(self):
        return "Sonido genérico"

class Perro(Animal):
    def hacer_sonido(self):
        return "Ladrido"

class Gato(Animal):
    def hacer_sonido(self):
        return "Maullido"

# Uso del polimorfismo
animales = [Perro(), Gato(), Animal()]
for animal in animales:
    print(animal.hacer_sonido())

# Salida:
# Ladrido
# Maullido  
# Sonido genérico

Importancia del Polimorfismo

  • Flexibilidad: Nuevas clases implementan funcionalidades sin modificar código existente
  • Reutilización: Métodos genéricos en diferentes tipos de objetos
  • Escalabilidad: Software adaptable a cambios y expansiones futuras

Composición: Una Alternativa a la Herencia

La composición establece relaciones "tiene un" sin crear jerarquías rígidas, promoviendo mayor modularidad y flexibilidad.

Diferencias Conceptuales

Herencia ("es un")

Relación padre-hijo

  • Subclase hereda atributos y métodos
  • Ejemplo: "Un Perro es un Animal"
  • UML: flecha con triángulo
  • Jerarquía clara y rígida
Composición ("tiene un")

Relación parte-todo

  • Clase contiene instancias de otra
  • Ejemplo: "Un Coche tiene un Motor"
  • UML: rombo negro
  • Flexibilidad y modularidad
Comparación en UML:
HERENCIA:                    COMPOSICIÓN:
+------------+               +------------+
|   Animal   |               |   Coche    |
+------------+               +------------+
|+hacer_sonido()              | - motor: Motor |
+------------+               +------------+
     ▲                            ♦
     │                            │
+------------+               +------------+
|   Perro    |               |   Motor    |
+------------+               +------------+
|+hacer_sonido()              | + encender() |
+------------+               +------------+
Implementación de composición:
class Motor:
    def encender(self):
        return "Motor encendido"

class Coche:
    def __init__(self):
        self.motor = Motor()  # Composición: Coche "tiene un" Motor
    
    def arrancar(self):
        return self.motor.encender()

mi_coche = Coche()
print(mi_coche.arrancar())  # Salida: Motor encendido

Casos de Uso de la Composición

Relaciones "Tiene un" (Has-a)

Entidad contiene otra entidad

  • Coche tiene Motor
  • Computadora tiene Procesador
  • Pedido tiene Productos
class Motor:
    def __init__(self, tipo):
        self.tipo = tipo
    def encender(self):
        return f"Motor {self.tipo} encendido"

class Coche:
    def __init__(self, marca, tipo_motor):
        self.marca = marca
        self.motor = Motor(tipo_motor)  # Composición
    
    def arrancar(self):
        return f"{self.marca}: {self.motor.encender()}"

Cuándo Usar Herencia vs Composición

Usar Herencia Cuando:
  • Existe relación "es un"
  • Clases comparten atributos comunes
  • Se requiere polimorfismo
  • Extender sin modificar clase base
Usar Composición Cuando:
  • No hay relación "es un" real
  • Se necesita flexibilidad
  • Evitar jerarquías rígidas
  • Bajo acoplamiento preferido

Patrones de Diseño: Principios y Aplicaciones

Los patrones de diseño son soluciones reutilizables a problemas comunes en el diseño de software.

¿Qué es un Patrón de Diseño?

Estrategia de programación que ofrece solución general a problema recurrente. No es código específico, sino guía estructurada.

Beneficios:

  • Reutilización de código
  • Comunicación entre desarrolladores
  • Mejoran mantenibilidad
  • Lenguaje común estándar

Clasificación de Patrones

Creacionales

Creación de objetos eficiente

  • Singleton
  • Factory Method
  • Builder
  • Abstract Factory
Estructurales

Relación entre clases/objetos

  • Adapter
  • Composite
  • Decorator
  • Facade
Comportamiento

Comunicación entre objetos

  • Observer
  • Strategy
  • Command
  • Template Method

Patrón Singleton

Una Única Instancia Global

Garantiza que una clase tenga una única instancia y proporciona acceso global

class Singleton:
    _instancia = None  # Variable de clase
    
    def __new__(cls):
        if cls._instancia is None:
            cls._instancia = super(Singleton, cls).__new__(cls)
        return cls._instancia

# Uso del Singleton
obj1 = Singleton()
obj2 = Singleton()
print(obj1 is obj2)  # Salida: True

Casos de uso: Gestión configuración, conexiones BD, logging

Patrón Factory

Creación Centralizada de Objetos

Delega responsabilidad de creación a método especializado

class Vehiculo:
    def conducir(self):
        pass

class Coche(Vehiculo):
    def conducir(self):
        return "Conduciendo un coche"

class Moto(Vehiculo):
    def conducir(self):
        return "Conduciendo una moto"

class VehiculoFactory:
    @staticmethod
    def crear_vehiculo(tipo):
        if tipo == "coche":
            return Coche()
        elif tipo == "moto":
            return Moto()
        else:
            raise ValueError("Tipo no válido")

# Uso del Factory
vehiculo1 = VehiculoFactory.crear_vehiculo("coche")
print(vehiculo1.conducir())  # Salida: Conduciendo un coche

Singleton vs Factory

Singleton: Una única instancia global, recursos compartidos

Factory: Múltiples instancias, creación flexible sin acoplamiento
