Interfaces de Usuario y Usabilidad

Fundamentos del diseño de interfaces centradas en el usuario

1. Introducción

Las Interfaces de Usuario (UI) son el punto de contacto entre los humanos y las computadoras. La usabilidad determina qué tan efectiva, eficiente y satisfactoria es esa interacción.

Concepto Central: Una interfaz no es solo lo que se ve, sino cómo se comporta y cómo el usuario la experimenta.
¿Qué estudiaremos?
  • Tipos y evolución de interfaces
  • Principios fundamentales de usabilidad
  • Métodos de evaluación
  • Métricas y herramientas
Objetivo

Comprender cómo crear interfaces que sean intuitivas, eficientes y agradables de usar.

2. Definiciones Clave

Interfaz de Usuario (UI)

Definición: El conjunto de elementos a través de los cuales las personas interactúan con un dispositivo o aplicación.

Incluye: Elementos visuales, controles, retroalimentación, comportamiento del sistema.

Usabilidad

ISO 9241-11: "Grado en que un producto puede ser usado por usuarios específicos para lograr objetivos específicos con efectividad, eficiencia y satisfacción."

Experiencia de Usuario (UX)

La experiencia total que tiene una persona al usar un producto, incluyendo aspectos emocionales, perceptuales y de comportamiento.

Accesibilidad

Capacidad de una interfaz para ser utilizada por personas con diferentes habilidades y en diferentes contextos.

3. Tipos de Interfaces

Interfaces de Línea de Comandos (CLI)

Características: Interacción basada en texto, comandos escritos, alta eficiencia para usuarios expertos.

Ventajas: Rápida, precisa, automatizable, consume pocos recursos.

Desventajas: Curva de aprendizaje alta, no intuitiva para principiantes.

$ ls -la
$ git commit -m "Update"
$ npm install
Interfaces Gráficas (GUI)

Características: Elementos visuales, mouse, ventanas, iconos, menús (paradigma WIMP).

Ventajas: Intuitiva, fácil de aprender, retroalimentación visual inmediata.

Desventajas: Puede ser menos eficiente para tareas repetitivas.

Windows, Icons, Menus, Pointer

Interfaces Táctiles (Touch)

Características: Interacción directa con los dedos, gestos, multitouch.

Ventajas: Natural, directa, no requiere dispositivos adicionales.

Gestos comunes: Tap, swipe, pinch, rotate, long press.

Smartphones, Tablets, Kioscos

Interfaces de Voz (VUI)

Características: Comandos hablados, respuestas audibles, procesamiento de lenguaje natural.

Ventajas: Manos libres, accesible, natural para humanos.

Ejemplos: Alexa, Siri, Google Assistant, sistemas de navegación por voz.

Asistentes Virtuales

4. Usabilidad

Definición de Jakob Nielsen

"La usabilidad es un atributo de calidad que evalúa qué tan fácil de usar son las interfaces de usuario."

Se compone de 5 componentes clave:

1. Aprendizaje (Learnability)

¿Qué tan fácil es para los usuarios realizar tareas básicas la primera vez que encuentran el diseño?

Métrica: Tiempo para completar una tarea por primera vez
2. Eficiencia (Efficiency)

Una vez que los usuarios han aprendido el diseño, ¿qué tan rápido pueden realizar tareas?

Métrica: Tiempo para completar tareas por usuario experimentado
3. Recordabilidad (Memorability)

Cuando los usuarios regresan al diseño después de un período sin usarlo, ¿qué tan fácil pueden restablecer la competencia?

Métrica: Tiempo para retomar eficiencia después de ausencia
4. Errores (Errors)

¿Cuántos errores cometen los usuarios, qué tan severos son estos errores y qué tan fácil se recuperan de los errores?

Métrica: Número y gravedad de errores, tiempo de recuperación
5. Satisfacción (Satisfaction)

¿Qué tan agradable es usar el diseño?

Métrica: Encuestas de satisfacción, análisis emocional, retención de usuarios

5. Principios de Usabilidad

Las 10 Heurísticas de Nielsen
1. Visibilidad del Estado del Sistema

El sistema siempre debe mantener informados a los usuarios sobre qué está ocurriendo, a través de retroalimentación apropiada dentro de un tiempo razonable.

Ejemplo: Barras de progreso en descargas, indicadores de carga, breadcrumbs de navegación.
2. Correspondencia entre el Sistema y el Mundo Real

El sistema debe hablar el idioma de los usuarios, con palabras, frases y conceptos familiares, en lugar de jerga del sistema.

Ejemplo: Íconos de carpetas para organización, papelera de reciclaje para eliminar archivos.
3. Control y Libertad del Usuario

Los usuarios a menudo eligen funciones del sistema por error y necesitarán una "salida de emergencia" claramente marcada.

Ejemplo: Botones de "Deshacer" y "Rehacer", cancelar procesos en curso.
4. Consistencia y Estándares

Los usuarios no deberían tener que preguntarse si diferentes palabras, situaciones o acciones significan lo mismo.

Ejemplo: Mismos iconos para mismas funciones, ubicaciones consistentes de menús.
5. Prevención de Errores

Mejor que buenos mensajes de error es un diseño cuidadoso que evite que ocurra un problema en primer lugar.

Ejemplo: Validación en tiempo real, confirmaciones antes de eliminar, deshabilitar opciones no válidas.

7. Evaluación de Usabilidad

Métodos con Usuarios
Testing de Usabilidad
  • Thinking Aloud: Usuario verbaliza mientras usa el sistema
  • Task-based Testing: Tareas específicas cronometradas
  • A/B Testing: Comparación entre versiones
  • Guerrilla Testing: Pruebas rápidas en espacios públicos
Observación
  • Eye Tracking: Seguimiento ocular
  • Analytics: Comportamiento real de usuarios
  • Heatmaps: Mapas de calor de interacción
Métodos sin Usuarios
Evaluación Heurística
  • Inspección por expertos usando las 10 heurísticas
  • 3-5 evaluadores independientes
  • Identificación de problemas y clasificación por severidad
Cognitive Walkthrough
  • Simula el proceso cognitivo del usuario
  • Pregunta clave: "¿El usuario sabrá qué hacer?"
  • Enfoque en aprendizaje por exploración
Análisis de Tareas
  • Descomposición de tareas complejas
  • Identificación de cuellos de botella

8. Metodologías de Evaluación

Evaluación Formativa

Cuándo: Durante el proceso de diseño

Objetivo: Mejorar el diseño

  • Prototipos de baja fidelidad
  • Sketches y wireframes
  • Pruebas iterativas frecuentes
  • Feedback rápido y cambios ágiles
Evaluación Sumativa

Cuándo: Al final del desarrollo

Objetivo: Validar el diseño final

  • Producto completamente funcional
  • Métricas cuantitativas precisas
  • Comparación con benchmarks
  • Certificación de usabilidad
Proceso de Evaluación Recomendado
Planificación
Definir objetivos, métricas, usuarios objetivo
Preparación
Crear escenarios, tareas, ambiente de prueba
Ejecución
Realizar pruebas, recopilar datos
Análisis
Interpretar resultados, priorizar problemas

10. Casos de Estudio

Caso de Éxito: Amazon
Problema Original

Proceso de compra complejo, carritos abandonados, navegación confusa.

Soluciones Implementadas
  • One-Click Purchase: Compra en un solo clic
  • Recomendaciones: "Customers who bought this..."
  • Reviews: Opiniones de otros usuarios
  • Wishlist: Guardar para después
Resultados
35% aumento en conversiones
25% reducción en abandonos de carrito
Caso de Fracaso: Microsoft Clippy
Intención Original

Asistente inteligente para ayudar a usuarios de Office con sugerencias proactivas.

Problemas Identificados
  • Interrupciones: Aparecía sin ser solicitado
  • Sugerencias irrelevantes: No entendía el contexto
  • Dificultad para desactivar: Proceso complejo
  • Ocupaba espacio: Interfaz visual intrusiva
Lecciones Aprendidas
El usuario debe tener control
La ayuda debe ser contextual
Menos puede ser más

11. Buenas Prácticas

Qué SÍ hacer
  • Conoce a tus usuarios: Investigación previa
  • Mantén la simplicidad: "Don't make me think"
  • Sé consistente: Mismos patrones, mismas funciones
  • Proporciona feedback: El usuario debe saber qué pasa
  • Permite errores: Deshacer y rehacer
  • Usa convenciones: No reinventes lo que funciona
  • Agrupa relacionados: Ley de proximidad
  • Jerarquía visual: Guía la atención
Qué NO hacer
  • No sobrecargues: Evita la saturación visual
  • No uses jerga técnica: Lenguaje del usuario
  • No ocultes funciones importantes: Visibilidad
  • No uses colores únicamente: Accesibilidad
  • No ignores estándares: Convenciones establecidas
  • No diseñes para ti: Diseña para el usuario
  • No omitas validación: Prueba siempre
  • No asumas conocimiento: Guía al usuario
Regla de Oro

"El usuario siempre tiene la razón, incluso cuando no la tiene." Si los usuarios no pueden usar tu interfaz, el problema es de la interfaz, no de los usuarios.
